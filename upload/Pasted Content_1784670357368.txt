# 2. REGRAS ABSOLUTAS

## 2.1. Não utilizar mocks

É proibido:

* criar arrays hardcoded que representem Merchants, Stores, Wallets, Payouts, Transactions ou Settlements;
* utilizar `setTimeout` para simular chamadas;
* criar rotas Next.js locais que imitem o backend;
* mostrar KPIs inventados;
* mostrar notificações fictícias;
* criar saldos bancários fictícios;
* criar Cases de consultoria fictícios;
* assumir sucesso financeiro antes da resposta do backend.

Quando não existir API real:

* mostrar empty state;
* mostrar “Coming Soon”;
* esconder a ação através de feature flag;
* documentar a limitação no relatório final.

## 2.2. O backend é a autoridade

O frontend nunca deve decidir por conta própria:

* se um utilizador é Admin;
* se um Merchant pode solicitar Payout;
* se o KYC é suficiente;
* se um saldo está disponível;
* se uma transição de estado é válida;
* se um Settlement pode ser libertado;
* se um Payout pode ser marcado como pago.

A interface pode esconder ações inválidas, mas o backend permanece a autoridade final.

## 2.3. Não alterar contratos silenciosamente

Quando a resposta real da API divergir do tipo TypeScript existente:

1. registar a resposta real;
2. adaptar o tipo;
3. adaptar o mapper;
4. documentar a divergência;
5. não mascarar o problema usando `any`.

# 5. SEMÂNTICA HTTP OBRIGATÓRIA

## 401 Unauthorized

Significa:

* token ausente;
* token inválido;
* token expirado;
* sessão já não válida.

Comportamento:

1. limpar access token;
2. limpar sessão persistida;
3. limpar dados sensíveis em memória;
4. redirecionar para `/login`;
5. preservar opcionalmente a rota de destino.

## 403 Forbidden

Significa:

* sessão válida;
* utilizador sem autorização para aquela operação.

Comportamento:

* não limpar o token;
* não redirecionar para login;
* mostrar página ou mensagem `Access denied`;
* permitir regressar à área Merchant.

## 404 Not Found

Não transformar automaticamente em empty state.

Distinguir:

* endpoint inexistente;
* recurso inexistente;
* feature ainda não implementada.

## 409 Conflict

Mostrar erro operacional específico.

Exemplos:

* saldo insuficiente;
* idempotency conflict;
* transição de estado inválida;
* operação já aplicada.

## 422 ou 400

Mostrar erros de validação junto aos respetivos campos sempre que possível.

## 429

Mostrar rate limit e permitir retry controlado.

## 500/502/503

Mostrar estado de indisponibilidade e botão Retry.

Não apagar sessão automaticamente.

# 7. FLUXO DE ARRANQUE DA APLICAÇÃO

Implementar esta sequência:

```text
1. Ler access token do storage
2. Token ausente → /login
3. Token presente → GET /auth/me
4. Sessão válida → GET /platform/capabilities
5. Depois → GET /platform/bootstrap
6. Hidratar Merchant, Workspace, Stores e features
7. Renderizar o produto permitido
```

Em caso de falha:

```text
/auth/me 401 → limpar sessão
/capabilities 403 → Merchant válido sem Admin
/bootstrap 500/502/503 → manter sessão e mostrar Retry
```

Evitar múltiplas chamadas simultâneas duplicadas durante a hidratação.
