    59тЖТ## Overview
    60тЖТ
    61тЖТ**XPAY.Expert** is a production-grade SaaS merchant dashboard for a multi-currency payment processing platform. It provides a comprehensive web interface for merchants to manage payments, wallets, settlements, payouts, risk monitoring, developer tools (API keys, webhooks), and banking services тАФ all powered by a remote REST API at `https://api.xpay.expert/api/v1`.
    62тЖТ
    63тЖТThe dashboard is a **frontend-only** application (no direct database connections). All data flows through authenticated API calls to the XPAY.Expert backend.
    64тЖТ
    65тЖТ### Design Principles
    66тЖТ
    67тЖТ- **Zero persisted secrets** тАФ JWT, full API keys, OTP codes, and action tokens are held in memory only; never written to `localStorage` or cookies.
    68тЖТ- **Security-first routing** тАФ route guards for merchant and admin areas; 401 triggers session invalidation, 403 preserves it.
    69тЖТ- **Server-driven capabilities** тАФ feature availability (banking, advisory, etc.) is determined by backend `capabilities` flags, with local static fallbacks.
    70тЖТ- **Mobile-first responsive** тАФ all views are designed mobile-first with progressive enhancement for desktop.

   186тЖТ### API Layer
   187тЖТ
   188тЖТThe API layer is centralised in `src/lib/api/` with three client configurations:
   189тЖТ
   190тЖТ| Client | File | Purpose |
   191тЖТ|---|---|---|
   192тЖТ| **public-client.ts** | Unauthenticated Axios instance | Login, registration, public endpoints |
   193тЖТ| **private-client.ts** | JWT-authenticated Axios instance | All authenticated merchant/admin endpoints |
   194тЖТ| **client.ts** | Legacy client | Backward compatibility |
   195тЖТ| **endpoints.ts** | Typed endpoint functions | All API calls (auth, commerce, banking, developer, admin) |
   196тЖТ| **xpApi.ts** | Extended API utilities | Convenience wrappers |

   198тЖТ| **mock.ts** | Mock data generators | Development & demo data |
   199тЖТ
   200тЖТ**Private client interceptor behaviour:**
   201тЖТ
   202тЖТ```typescript
   203тЖТ// JWT injected from in-memory reference тАФ never persisted
   204тЖТprivateApi.interceptors.request.use((config) => {
   205тЖТ  if (_accessToken) config.headers.set("Authorization", `Bearer ${_accessToken}`);
   206тЖТ  return config;
   207тЖТ});
   208тЖТ
   209тЖТ// 401 тЖТ clear session immediately, redirect to /login
   210тЖТ// 403 тЖТ preserve session (access denied, not a session issue)
   211тЖТ```

   237тЖТ## API Integration
   238тЖТ
   239тЖТ### Backend API
   240тЖТ
   241тЖТAll requests target the XPAY.Expert REST API:
   242тЖТ
   243тЖТ```
   244тЖТBase URL: https://api.xpay.expert/api/v1
   245тЖТ```
   246тЖТ
   247тЖТThe frontend communicates exclusively with this API. There are no direct database connections or server-side data fetching from external sources.
   248тЖТ
   249тЖТ### API Clients
   250тЖТ
   251тЖТThree Axios instances handle different authentication contexts:
   252тЖТ
   253тЖТ**Public Client** тАФ No authentication header. Used for login, register, and public endpoints.
   254тЖТ
   255тЖТ**Private Client** тАФ Injects `Authorization: Bearer <token>` on every request from an in-memory variable. Handles:
   256тЖТ- 401 тЖТ clears the in-memory token, calls the registered unauthorized handler (redirects to `/login`)
   257тЖТ- 403 тЖТ preserves session, rejects with access-denied error
   258тЖТ- Network errors тЖТ preserved session, user can retry
   259тЖТ
   260тЖТ**Security-Aware Client** тАФ Extends the private client with the `X-Security-Action` header for sensitive operations (API key creation, rotation, transfer confirmation, email verification completion).
   261тЖТ
   262тЖТ### Response Envelope
   263тЖТ
   264тЖТAll authenticated endpoints return a standardised envelope:
   265тЖТ
   266тЖТ```json
   267тЖТ{
   268тЖТ  "success": true,
   269тЖТ  "data": { ... },
   270тЖТ  "error": { "code": "ERROR_CODE", "message": "Human-readable message" }
   271тЖТ}
   272тЖТ```
   273тЖТ
   274тЖТ`privateRequestData<T>()` automatically unwraps the envelope and returns `T`. If `success` is `false`, it throws a normalised `ApiError`.

   377тЖТ## Security Model
   378тЖТ
   379тЖТ### Authentication Flow
   380тЖТ
   381тЖТ1. User submits email + password to `POST /auth/login`
   382тЖТ2. Server returns `{ token, merchant }` тАФ JWT access token + merchant profile
   383тЖТ3. Frontend stores token **in memory only** (Zustand store + Axios interceptor variable)
   384тЖТ4. On page refresh, the Zustand `persist` middleware rehydrates the token from `localStorage` into memory, then validates with `GET /auth/me`
   385тЖТ5. Invalid tokens are immediately cleared on any 401 response
   386тЖТ
   387тЖТ### JWT Lifecycle
   388тЖТ
   389тЖТ| Event | Behaviour |
   390тЖТ|---|---|
   391тЖТ| **Login** | Token set in memory and Zustand store (persisted to localStorage) |
   392тЖТ| **Page refresh** | Token rehydrated from localStorage тЖТ validated via `GET /auth/me` |
   393тЖТ| **401 response** | In-memory token nulled, Zustand cleared, redirect to `/login` |
   394тЖТ| **403 response** | Session preserved (access denied тАФ not a session issue) |
   395тЖТ| **Logout** | Server-side invalidation + client-side memory/storage clear |
   396тЖТ| **Network error** | Session preserved тАФ user can retry |
   397тЖТ
   398тЖТ> **No refresh tokens.** Sessions are short-lived. A 401 always means re-authentication is required.
   399тЖТ
   400тЖТ### Error Handling
   401тЖТ
   402тЖТThe private client normalises all Axios errors into a consistent `ApiError` shape:
   403тЖТ
   404тЖТ```typescript
   405тЖТinterface ApiError {
   406тЖТ  message: string;  // Human-readable error description
   407тЖТ  code?: string;   // Machine-readable error code
   408тЖТ  status?: number;  // HTTP status code
   409тЖТ}
   410тЖТ```
   411тЖТ
   412тЖТ| HTTP Status | Derived Code | UI Behaviour |
   413тЖТ|---|---|---|
   414тЖТ| `401` | тАФ | Clear session, redirect to login |
   415тЖТ| `403` | тАФ | Preserve session, show "Access denied" |
   416тЖТ| `409` | `CONFLICT` | Show "Resource conflict" message |
   417тЖТ| `429` | `RATE_LIMITED` | Show "Too many requests тАФ please wait" |
   418тЖТ| `503` | `SERVICE_UNAVAILABLE` | Show "Service temporarily unavailable" |
   419тЖТ
   420тЖТ### API Keys v2
   421тЖТ
   422тЖТAPI keys follow a strict reveal-once policy:
   423тЖТ
   424тЖТ- **`GET /developer/api-keys`** тАФ returns key metadata (prefix, last 4 characters, scopes, status). The `fullKey` is **never** returned.
   425тЖТ- **`POST /developer/api-keys`** тАФ returns the `fullKey` in the response. This is the **only time** the full key is visible. It must be copied by the user immediately.
   426тЖТ- **`POST /developer/api-keys/:id/rotate`** тАФ invalidates the old key, returns a new `fullKey` once. Same reveal-once policy.
   427тЖТ- **`POST /developer/api-keys/:id/revoke`** тАФ permanently disables the key.
   428тЖТ
   429тЖТThe `fullKey` is **never persisted** тАФ not in state, not in storage, not in logs. Once the user leaves the creation/rotation screen, the key is gone.
   430тЖТ
   431тЖТ### Webhooks
   432тЖТ
   433тЖТWebhook management is split across two API generations:
   434тЖТ
   435тЖТ**Legacy** (still active):
   436тЖТ- Full CRUD at `GET/POST/PATCH/DELETE /webhooks`
   437тЖТ
   438тЖТ**Merchant v2**:
   439тЖТ- CRUD at `/merchant/webhooks` with delivery health metrics (`last24h`, `avgLatencyMs`)
   440тЖТ- Secret rotation via `POST /merchant/webhooks/:id/rotate-secret` (requires security action)
   441тЖТ
   442тЖТ> **Note:** Provider v2 webhook endpoints (`/provider/webhooks`) are read-only. Full Provider v2 CRUD is not yet available.
   443тЖТ
   444тЖТ### Security Challenges
   445тЖТ
   446тЖТSensitive operations (API key creation, key rotation, transfer confirmation, email verification) require a completed security challenge:
   447тЖТ
   448тЖТ1. Client calls `GET /security/purposes` to determine the challenge type
   449тЖТ2. Client calls `POST /security/challenges/request` тЖТ triggers a **6-digit OTP via email**
   450тЖТ3. User enters the OTP тЖТ client calls `POST /security/challenges/verify`
   451тЖТ4. Server returns an `actionToken`
   452тЖТ5. `actionToken` is passed as the `X-Security-Action` header in the sensitive request
   453тЖТ6. `actionToken` is held **in memory only** тАФ never persisted
   454тЖТ

   671тЖТ- **Component**: Language switcher component at `src/components/shared/language-switcher.tsx`
   672тЖТ
   673тЖТ```typescript
   674тЖТ// Usage in components
   675тЖТimport { useT } from "@/lib/i18n";
   676тЖТ
   677тЖТfunction MyComponent() {
   678тЖТ  const t = useT();
   679тЖТ  return <h1>{t("nav.dashboard")}</h1>;
   680тЖТ}
   681тЖТ```
   682тЖТ
   683тЖТThe system respects persisted user choices тАФ if a user has already selected a locale, automatic browser detection will not override it.
